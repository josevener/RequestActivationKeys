# AI_AGENT.md

**Project:** Web Approval App for Activation Key Requests
**Architecture:** React (Vite) + Node.js + Existing Main System Database
**Auth Source:** `tblSecurityUsers` (existing DB)
**Access Scope:** Marvin, Sharlyn, Diana only
**Database:** Reuse existing schema and approval logic (no migrations)

---

## 1. Purpose of This Agent

This AI Agent acts as a technical implementation guide and validation assistant for building a **standalone web approval module** for Activation Key Requests that:

* Uses the same database as the main system
* Uses the same user credentials (LDAP already handled on login page)
* Is accessible outside VPN
* Is mobile-friendly
* Does NOT allow create/edit
* Strictly mirrors main system approval logic

## 5. Backend Specification

### Base Route

```
/api/activation-key-requests
```

---

### 5.1 GET Pending Requests

```
GET /api/activation-key-requests?status=pending
```

#### Behavior

* Return only requests not yet approved.
* Must match main system's definition of "pending".
* Join header + details table as needed.
* Only return required fields.

#### Response Shape

```json
[
  {
    "request_id": 101,
    "days_trial": 14,
    "employee_count": 50,
    "permanent": true,
    "unli_emp": false,
    "opt_date": "2026-03-03",
    "system_edition": "System X"
  }
]
```

#### Required Fields (Strictly These Only)

* DaysTrial
* EmployeeCount
* IsPermanent
* IsUnlimitedEmployeeCount
* OptimizationDate
* SystemEdition

No editing fields.
No internal flags.
No audit fields.

---

### 5.2 Approve Endpoint

```
POST /api/activation-key-requests/approve
```

#### Request Body

```json
{
  "request_ids": [101, 102, 103]
}
```

#### Validation

| Case                | Result |
| ------------------- | ------ |
| Missing request_ids | 400    |
| Empty array         | 400    |
| Non-array           | 400    |

## 7. User Flow

1. User logs in.
2. Backend validates:

   * Credentials
   * Username in allowlist
3. Frontend loads pending requests.
4. User selects:

   * Single Approve
   * OR multiple + Approve Selected
5. Backend processes.
6. List refreshes.

## 8. Edge Case Handling

### Invalid Credentials

Return 401.

### Valid but Not Allowed

Return 403.

### Approve Empty Array

Return 400.

### Race Condition

If request already approved:

Behavior must match main system:

* Either:

  * Skip and return partial
  * Or fail entire transaction

Developer must inspect existing behavior.

## 9. Concurrency Requirements

Approval must be protected by:

* Transaction
* Row-level lock (if main system uses it)
* OR conditional update:

Example pattern:

```sql
UPDATE tblActivationKeyRequestDetails
SET ApprovedById = @UserId,
    ApprovalDate = GETDATE()
WHERE Id = @Id
AND ApprovedById IS NULL
```

This prevents double approval.
